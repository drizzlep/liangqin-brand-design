# 良禽佳木生成执行清单

用于把 `DESIGN.md` 的抽象品牌规则翻译成生成前、中、后的显式自检问题。

## 1. 生成前检查

- [ ] 是否已先读取 `DESIGN.md`，而不是直接从参考图或历史样例开始
- [ ] 是否已完成 Context Gate：品牌 brief、产品事实、参考材料、真实资产和目标保真度是否足够
- [ ] 是否已完成 Surface Gate：这次交付是网页、H5、报价卡、PPT、原型、画布、动效还是独立 HTML
- [ ] 是否明确了这次任务的页面目标、核心内容和服务动作
- [ ] 是否把这次变化理解为内容变化或承载变化，而不是风格切换
- [ ] 如果需要变体，是否只在 `design-packs/` 的受控 tweak 轴内探索
- [ ] 若产品事实不足，是否准备输出低细节结构，而不是用装饰补足“完成感”
- [ ] 若图片、图标、报价、logo 或组件不足，是否准备使用 placeholder、待补资料清单或停在 spec
- [ ] 是否已确认中文优先、礼貌 CTA、摄影与材质优先仍是本次生成的硬规则

## 2. 生成中检查

- [ ] 当前结果是否仍然像良禽佳木，而不是泛高端家居品牌
- [ ] 中文标题、按钮、标签和说明是否自然完整，而不是英文 UI 或翻译腔
- [ ] 是否仍然由摄影、材质、留白和结构承担主气质，而不是图标和装饰承担主气质
- [ ] CTA 是否仍然礼貌、邀请式，没有催促感和成交压迫感
- [ ] 图标是否符合细线、几何、克制的工艺要求；如果做不到，是否应直接减少或取消
- [ ] 是否没有用无意义数字、假标签、假卖点或多余图标制造“丰富感”
- [ ] 是否没有把非网页交付物硬套成网页落地页结构
- [ ] 是否遵守当前 artifact surface 的画幅、交互、导出和验证合同
- [ ] 是否出现了 `Pack`、`Page Goal`、`Preview`、`Design Preview` 等系统说明词

## 3. 生成后检查

- [ ] 这份输出是否像真实成品，而不是规范说明卡或系统演示图
- [ ] 去掉图片后，仅看文字和结构，是否依然成立
- [ ] 如果换一个品类，这个结果是否仍会被认成同一品牌
- [ ] 如果换一个承载，这个结果是否仍会被认成同一品牌的不同交付
- [ ] 是否存在低质量图标、过度卡片化、说明文泄漏、英文元标签等明显失真项
- [ ] 如某些细节工艺无法稳定成立，是否已经优先做了降级，而不是自由发挥
- [ ] 是否完成结构、运行、视觉和交付四类验证；若没有，是否明确说明当前只是一版 spec 或草稿
- [ ] 是否明确当前交付状态：structured spec、视觉草稿、可交互原型、最终成品或待补资料阻断
